# batch_lint

目的：

针对指定 library 的 `wiki/sources/` 目录下全部 `sources` 文件，逐个执行 `lint`，完成链接检查与双向链接补充。

命令格式：

- `batch_lint [libraryName]`

示例：

- `batch_lint coreWiki`

输入：

- 一个 `libraryName`

输入要求：

- 必须显式提供 `libraryName`，不能省略。
- 必须只针对 `[libraryName]/wiki/sources/` 目录执行，不得扩展到 `wiki/practice/`、`wiki/query/` 或其他目录。

执行要求：

1. 必须先枚举 `[libraryName]/wiki/sources/` 下的所有 markdown 文件。
2. 必须对枚举到的每一个文件逐个执行完整的 `lint` 流程。
3. 每个文件的检查标准、链接判断标准、双向链接补充方式、输出要求与硬性约束，都必须完全遵循 `script/commands/lint.md`，不得自行简化。
4. 执行 `batch_lint` 时，不得因为是批量处理而跳过对单个目标文件的完整阅读，也不得跳过对候选旧 `sources` 文件的详细阅读。
5. 批量处理过程中，后续文件必须基于前面文件已经产生的最新 `wiki/` 内容继续执行；不得把所有文件都当作彼此隔离的静态快照处理。
6. 如果某个文件检查后无需新增链接，也必须在结果中明确记录该文件已检查但未改动。
7. 如果 `[libraryName]/wiki/sources/` 下没有任何文件，必须明确说明本次没有可执行的目标文件。
8. 检查完成后，应输出本次批量处理过的文件列表、每个文件最终新增或未新增链接的结果，以及全库范围的矛盾与取代关系汇总（逐文件 lint 中按 `lint.md` 第 14、15 条检出并处置的条目）。
9. 概念候选检测（只报告，不修改）：批量 lint 本来就会精读全部 sources 文件，过程中必须顺带记录反复出现的概念信号（双向链接说明文字中的高频主题词、多个来源共同论及的知识单元或思维模型）。全部文件处理完成后，读取 `[libraryName]/wiki/concepts/index.md`（如存在），对照 `## 概念清单` 与 `## 候选概念`，在结果末尾输出一份"概念候选提示"：列出被 3 个及以上来源真实论及、但尚无概念页的概念（含涉及来源），并建议用户执行 `concept [libraryName]` 建页。本步骤不得新建概念页，概念页的建立只能由 `concept` 命令完成（逐文件 lint 过程中按 `lint.md` 第 14 条对概念页争议节的补写与张力候选登记不受本条限制）。
10. 覆盖缺口提示（只报告，不修改）：批量处理结束后，基于全库阅读所得，列出"被多个来源浅浅提及、但库内没有任何深入资料"的主题（对应 Karpathy lint 的数据缺口维度），建议用户补充新的 raw 来源。没有明显缺口时可省略本节，不得为凑数硬列。

硬性约束：

- 只允许修改 `[libraryName]/wiki/` 下的文件。
- 允许的修改范围与 `lint.md` 的硬性约束一致：补充双向链接、概念页争议节补写与张力候选登记、取代关系标注；除此之外不得修改。
- 禁止执行复制、剪切、移动或重命名 raw 文件的操作。
- 不允许伪造问题，也不允许忽略真实结构错误。
- 若判断不充分，必须标注 `Confidence: Low`。
- 涉及实际链接补充时，必须使用简体中文。

输出结果：

- 可能更新的相关 wiki 文件
- 一份简短批量检查结果
- 一份"概念候选提示"（如有达到成页门槛但尚无概念页的概念）
