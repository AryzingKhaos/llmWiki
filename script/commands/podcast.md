# podcast

## 基础要求

目的：

把一篇 wiki 摘要（`wiki/sources/*.md`）或一整本书 / 一份 raw 材料，改写成**适合"听"的口播稿**（口语化、耐断、能连续朗读），并可选地把口播稿转成音频。核心是"从'读的文字'变成'听的文字'"，不是逐字念摘要。

命令格式：

- `podcast [文件路径] [风格?]`

示例：

- `podcast investWiki/wiki/sources/202305-FurtherThoughtsOnSeaChange-霍华德马克斯.md`
- `podcast coreWiki/raw/2026/纳瓦尔宝典.pdf duo`

适用场景：

- 想把已有 wiki 摘要变成通勤 / 骑车能听的音频内容
- 想把一整本书 / 长 raw 材料转成"分集口播稿"来听
- 需要一份可交给 TTS 朗读的干净口语稿

输入：

- 一个文件路径（**唯一必填项**）。可以是 `wiki/sources/*.md`，也可以是 `raw/` 下的原始材料。
- 可选：风格（`solo` 默认 / `duo`）。

输入要求：

- **`libraryName` 从路径自行推断，不需要用户显式提供**：取路径中第一个形如 `xxxWiki/` 的目录段作为 `libraryName`（如 `investWiki/wiki/sources/...` → `investWiki`；`coreWiki/raw/...` → `coreWiki`）。只有当路径里找不到任何 `*Wiki/` 段、无法推断 library 时，才提示用户补充。
- `[文件路径]` 指向 `wiki/sources/`（推荐，已是压缩过的知识）或 `raw/`（原始材料，更全但更长）均可。若指向的文件在某个 library 的 `raw/` 之外、且不属于任何 library，必须先明确警告用户。
- 风格默认 `solo`；命令里带 `duo` 才走双人对谈。不需要询问用户风格、时长、音色——直接按默认值执行。

输出结果（默认同时产出文本 + 音频）：

- **每个产物独占一个文件夹**：`[libraryName]/wiki/podcast/[yyyyMM]-[中文名]-口播/`，所有产物文件都放进这个文件夹，不散落在 `wiki/podcast/` 根目录。文件夹内包含：
  - `口播稿.md`（口播稿正文）
  - `口播-云希.mp3`（整集音频；长材料分集时为 `口播-云希-01.mp3` …；音色变化时文件名末尾换成对应音色名）
  - 可选 `音色小样/`（做音色对比时才有，存放各音色试听小样）
- 更新 `[libraryName]/wiki/podcast/index.md`（口播稿索引，位于 `podcast/` 根目录，指向各产物文件夹；没有则新建）

## 声音说明（两层，务必区分）

口播的"声音"分两层，命令分别控制：

### 第一层：口播体裁（写法 / 口吻）——`风格`参数

- `solo`（**默认**）：**单人独白讲解**。一个主播娓娓道来，最耐断、最适合骑车 / 通勤听。适合论述型、备忘录、单一作者观点的材料。
- `duo`：**双人对谈**。主持人 + 嘉宾一问一答（类似 Google NotebookLM 的双人播客）。信息更活、有节奏感，但朗读时需要两个不同音色交替。适合观点碰撞、需要"提问—展开"的材料。
- `story`：**说书 / 评书体**。更戏剧化、有悬念铺陈。适合故事、历史、人物传记类材料。

### 第二层：TTS 音色（真正朗读的嗓音）

**默认总是生成音频，默认音色 `edge:云希`（zh-CN-YunxiNeural，沉稳男声，口播首选）**，无需任何开关。若用户在命令里显式点名了其它音色，才改用指定音色。可选后端：

- `edge:<voice>` —— **默认后端**，`edge-tts` 已安装，免费、音色自然、可选项多。常用中文音色：
  - `云希`（YunxiNeural，男声，沉稳，**默认**）
  - `晓晓`（XiaoxiaoNeural，女声，亲和）
  - `云扬`（YunyangNeural，男声，新闻播报腔）
  - `晓伊`（XiaoyiNeural，女声，年轻）
  - `duo` 风格默认配音：主持人 `晓晓` + 嘉宾 `云希`
- `say:<voice>` —— macOS 内置，零依赖，作为 `edge-tts` 不可用时的**降级后端**。中文音色：`Tingting`（婷婷·普通话）、`Meijia`（美佳·台湾）、`Sinji`（善怡·粤语）。
- `openai:<voice>` / `eleven:<voice>` —— 音质最好，需 API key（仅在用户显式指定时用；未配置则提示并降级）。

若默认的 `edge:云希` 因环境 / 网络不可用，必须**先明确告知缺什么、如何补**，并自动降级为 `say:Tingting` 出音频，不得静默失败。

## 执行要求

1. **先判断输入体量**：读取输入文件，统计正文字数。
   - 短材料（≤ 约 4000 中文字 / 2000 英文词）：产出**单集**口播稿（`口播稿.md` + `口播-云希.mp3`）。
   - 长材料（书、长 raw）：**按原有章节/课/篇切分，一章一集，每章单独一个 mp3**。若原文有清晰的目录 / 章节标记（如"第N章""第N课"），必须优先按它切分；没有则按主题切成每集 8–15 分钟。切分与分集属确定性工作，应先完成再改写。
2. **口语化改写，而非复述**。把"读的文字"改成"听的文字"，必须做到：
   - 去掉一切 markdown 结构噪声：表格、图（mermaid/dot）、`>` 引用符号、行内链接、wiki 交叉链接（`[[...]]` / `[..](..)`）、`Source Path`/`重点等级`/`Confidence` 等元信息行——这些都不进口播稿。
   - 把书面语连接词改成口语过渡：如"如下所示 / 综上 / 见下图" → "接下来说的是 / 我们捋一捋 / 打个比方"。
   - 引文处理：原文里的 `>` 引用要转成**口语化的转述 + 归属**，例如"马克斯打了个比方，他说……"。英文原句默认**只保留中文**（听不出括号里的英文）；确需保留英文金句时，用"原话是这么说的：……"引出，且必须能顺畅读出。
   - 补足"听感脚手架"：开场 30 秒的**引子/钩子**、段落间的**过渡句**、结尾的**收束总结**。这些是口播体裁的黏合剂，属"表达方式"而非"新事实"。
3. **忠于原文，禁止编造**。口播稿中的一切事实、数据、观点、人名、引文都必须在输入材料中真实存在；严禁补充材料中未出现的新事实、新数据、新结论。只允许改写口吻、增加过渡与开场结尾等**不含新信息**的表达性内容。（这条与 `ingest` 的忠实性约束一致。）
4. **篇幅由"输入类型"决定，不要盲目压缩**。按中文口播语速约 **240–300 字/分钟** 折算（目标时长 × 270 ≈ 目标字数）：
   - 输入是 `wiki/sources/*.md`（**已压缩过的摘要**）：口播稿与源摘要大致 1:1，做口语化展开即可，不再二次压缩。
   - 输入是 `raw/` 原始材料（书、逐字稿、讲义等**未压缩全文**）：**默认走"完整版·轻量删减"**——几乎保留原文全部实质内容（概念、研究、人名、故事、案例、数据、引文一个都不删），只去掉**不适合放进博客/不适合听**的东西：课堂管理话（问好、点名、作业、考试、休息、"下节课见"）、口水与重复、"看PPT/黑板/如图"这类视觉现场指代、排版噪声。正常情况下每集口播稿字数应接近对应原文（约原文 75%–95%），**不得**为了"精简/提炼精华"把一节课砍到原文的三分之一甚至更短。用户若显式要"精简版/摘要版"才做大幅压缩。
   - 用户给了 `[时长]` 就按它收敛；没给且输入为摘要时默认单集 8–12 分钟；输入为 raw 全文时以"讲完整"为先，时长随内容自然长短，不设上限。
5. **风格落地**：
   - `solo`：全程第一/第三人称叙述，一个声音讲到底。
   - `duo`：写成剧本式对白，用 `主持人：` / `嘉宾：`（或具名）领起每句；提问推进、回答展开，交替自然。朗读时两角色分配不同音色。
   - `story`：加入适度悬念铺陈与场景感，但事实仍受第 3 条约束。
6. **产物文件夹与口播稿结构**：
   - 先在 `wiki/podcast/` 下建产物文件夹 `[yyyyMM]-[中文名]-口播/`；`[yyyyMM]` 取当前月份，`[中文名]` 可沿用源材料名，文件夹重名时追加 `-2`、`-3`。
   - 口播稿固定命名为该文件夹内的 `口播稿.md`：
     - 顶部一个元信息块（**不朗读**，仅供人查阅）：源文件路径、体裁风格、预计时长/字数、使用音色、生成日期。
     - 正文即"可直接朗读的连续文本"。如为多集，用 `## 第N集·<小标题>` 分隔；集内不再堆二级标题，避免朗读时读出一堆标题。
     - 文件末尾附一段"音频生成记录"：用了哪个后端 / 音色 / 生成了哪些 mp3。
   - 音频与可选的 `音色小样/` 也都放进同一个产物文件夹，不外泄到 `podcast/` 根目录。
7. **索引**：更新 `wiki/podcast/index.md`（位于 `podcast/` 根），每条至少包含：指向产物文件夹内 `口播稿.md` 的链接、源文件路径、风格、时长、音频文件名、一句话简介。没有该文件则新建。
8. **主语言必须是简体中文**（粤语音色 `Sinji` 除外时也仍用简体中文文本，由 TTS 决定发音）。

## 询问规则

**默认不询问，直接按默认值执行**：`solo` 单人独白 + 单集 8–12 分钟（长材料自动分集，每集 8–15 分钟）+ 生成 `edge:云希` 音频。命令里带 `duo` 则改双人对谈。

只有以下情况才需要开口：

- 路径里找不到任何 `*Wiki/` 段、无法推断 `libraryName` —— 提示用户补充库名。
- 目标文件不存在、或在任何 library 的 `raw/` 之外且不属于任何 library —— 先警告再决定是否继续。

除此之外一律不打断，直接产出口播稿 + 音频。

## 音频生成（默认执行）

合成前必须先把口播稿的元信息块（HTML 注释）与集标题剥离，只喂正文纯文本。

1. **默认** `edge:云希`：`python3 -m edge_tts --voice zh-CN-YunxiNeural --file episode.txt --write-media out.mp3`。其它 edge 音色同理替换 `--voice`。
2. 降级 `say:<voice>`：`say -v <voice> -o out.aiff -f script.txt` 后用 `ffmpeg` 转 `mp3`。仅在 `edge-tts` / 网络不可用时启用，并提示用户。
3. `duo` 风格出音频：按角色切分文本，主持人 `晓晓` + 嘉宾 `云希` 分别合成分段，再用 `ffmpeg` 按顺序拼接为整集。
4. 长材料多集：每集单独一个 mp3，命名 `口播-云希-01.mp3` … 并在索引中登记。
5. 音频只写入当前产物文件夹 `wiki/podcast/[yyyyMM]-[中文名]-口播/` 内，不外泄到 `podcast/` 根、不触碰 `raw/`。

## 硬性约束

- 只允许修改 `[libraryName]/wiki/podcast/` 下的文件（口播稿、音频、索引）；不得改写 `wiki/sources/`、不得改动 `raw/`。
- 严禁虚构事实、数据、引文、人名或来源；口播稿只能对输入材料做"口语化改写 + 表达性黏合"，不得注入新信息。
- 主语言必须是简体中文。
- 默认音色 `edge:云希` 不可用时，必须显式说明缺什么、如何补，并自动降级为 `say:Tingting` 出音频，不得静默失败。
- 若输入指向 `raw/` 之外且不在本 library 下，必须先警告用户。
- 若本规范与运行环境的上层约束冲突，必须显式说明冲突点并询问用户，冲突未澄清前不得擅自绕过。
