# ingest

## 基础要求

目的：

将单个 raw 文件编译为结构化 wiki 知识。最重要的事情是：1.内容摘录；2.跟已有的wiki知识相链接

命令格式：

- `ingest [libraryName] [文件路径]`

示例：

- `ingest coreWiki coreWiki/raw/articles/example.md`

适用场景：

- 已有 raw 文件内容发生变化
- 需要临时导入不在 `[libraryName]/raw/` 下的单个文件，但必须先明确警告用户该文件不在 raw 目录下
- 手动触发单文件导入

输入：

- 一个 `libraryName`
- 一个文件路径

输入要求：

- 必须显式提供 `libraryName`，不能省略。如果省略，必须提示用户补充，否则不能进行之后的步骤。
- `[文件路径]` 可以不位于对应 library 的 `raw/` 目录下；但如果不在 `[libraryName]/raw/` 下，必须先明确警告用户该文档不在 raw 目录下，再继续后续流程。
- 如果用户尚未提供资料的重点等级，必须先进入“询问规则”；只有在用户未回复、拒绝回答或无法提供时，才将所有未定模块默认按 D 级处理。

输出结果：

- 新建或更新的 `[libraryName]/wiki/sources/<source>.md`
- 更新后的 `[libraryName]/wiki/index.md`
- 可能新建的 `[libraryName]/wiki/practice/<practice>.md`

## 执行要求

1. 创建摘要页面，每个输入文件理论上对应一个 `sources/` 下的摘要页面；将主要内容写到摘要页面中。重点等级按模块生效；在符合“重点等级划分”的前提下，对于信息密度比较高的文字，适当降低压缩比例；对于信息密度比较低的文字，适当提高压缩比例。无论是否为长文，每个模块对应的原始材料范围都不得超过5000中文字；如果模块主要使用英文书写，则对应的原始材料范围不得超过2500个英文单词。若超过该上限，必须在压缩前继续拆分为更小的子模块。
2. 在摘要中要总结原文的内容，划分抽象的概念、方法论、系统流程，以及具体的经验、亲身经历，对于抽象、复杂化的系统概念，可以用mermaid或者dot图解释，但不要滥用图片。对于具体的经验，可以大段引述原文内容。
3. 就关键问题与用户沟通。关于询问的规则，请见“询问规则”
4. 检索 index.md，确认可能有关的页面；在摘要压缩任务全部完成后，必须按“新入库后的双向链接检查规范”执行双向链接检查。
5. wiki分为两种，一种是对于 raw 文件的摘要，放在wiki/sources/ 下；另一种依赖“问题”，用户可以想要解决某个问题提出了一个问题（也就是使用query命令时），这个问题的回答也可以作为一个wiki页面，放在 wiki/query/ 下。这里仅对query行为做说明，ingest 不需要关心 query 的行为。
6. 关于原文摘录的限制：如果是原文中信息量很大的文字，或者非常经典的文字，可以适当摘抄到摘要中，通常每段文字的长度不超过300字。
7. 必须有原始文件的地址。原始文件地址字段统一使用格式：`Source Path: <相对仓库根目录的路径>`。如果源文件不在 `[libraryName]/raw/` 下，必须额外添加一行：`Warning: source file is outside [libraryName]/raw/`。
8. 主语言必须是简体中文。
9. 摘要中必须是原文件中明确存在的信息，不得捏造、搬运其他任何信息。严禁补充资料中未出现的新事实、新概念、新结论或额外背景（实践计划除外），即使输入资料中的相关概念非常少，也不能自行发挥补写内容。
10. 对于很长的内容（大于5000中文字的内容，或者大于2500个英文单词的内容），也只进行一次压缩，压缩完成后直接写入 `wiki/sources/`。`wiki/simplify/` 不再用于新的 ingest 流程。长文的具体执行方式见“长文压缩计划”。
11. `wiki/sources/` 下的文件命名规则是：`[yyyyMM]-[中文的名字(可以直接用原名)]`，如果重名，可以在最后增加“-2”、“-3”
12. 更新 `wiki/index.md` 时，必须按照“Index 示例格式说明”写入；除示例外的其他条目也统一参考该模块的格式。`index.md` 中不再记录 `Module Names` 和 `Module Levels`。
13. 生成到 `wiki/sources/` 下的 md 文件时，所有级别的标题都必须保证“标题内容”不含任何空格字符：
	- 适用于 `#` 到 `######` 的全部标题级别
	- 标题内容中不允许出现半角空格、全角空格或 Tab
	- 只允许保留 markdown 标题语法本身需要的前缀，例如 `## ` 或 `### `
	- 示例：应写成 `## A.第一章上半：聪明的投资者将获得什么+一战前的投资与当今投资之比较`，不要写成 `## A. 第一章上半：聪明的投资者将获得什么 + 一战前的投资与当今投资之比较`

### 询问规则
1. 需要先根据来源内容列出具体的模块。如果来源内容越短，那么需要划分的模块就越少。每个模块都应该有代号，比如“一.财富 二.幸福 三.责任感”等
2. 每个问题尽量根据内容给出全面的备选项。
3. 每个内容都应该根据重点等级划分，可以参考“重点等级划分”部分
4. 必须询问每个模块的重点等级，使用代号来描述等级，比如“1.财富 2.幸福 3.责任感”对应“1.A级 2.D级”，未提及的内容一律视为 D 级，也就是“1.A级 2.D级”对应实际的回答是“1.A级 2.D级 3.D级”；如果用户未回复、拒绝回答或无法提供，则所有未定模块默认视为 D 级。
5. 如果某个内容被判定为 A 级内容，需要进行第二轮询问，跟用户讨论编写实践计划的要求，关注点主要是：每天的实践时间、计划持续的时间长度等。
6. 在用户答完各模块的重点等级之后，AI 必须强制复述每个模块对应的压缩比例，明确说明该模块在摘要压缩中的比例，不允许省略。

### 重点等级划分
1. 内容分为 A-E 级：
	0. 重点等级按模块生效，不按整篇文档统一生效。
	1. A级内容指“必须详细了解而且要实践”的内容，摘要压缩比例在1/2到1/4，必须使用md的引用添加这部分的核心内容、信息密度高的内容，必须在 wiki/practice/ 文件夹下创建专门的实践计划，实践计划如何写请参考“实践计划”部分
	2. B级内容指“必须详细了解，但是当前还没有时间实践”的内容，摘要压缩比例在1/2到1/4，必须使用md的少量引用添加这部分的核心内容、信息密度高的内容。
	3. C级内容指“需要重点了解，但不考虑实践”的内容，摘要压缩比例在1/4到1/8，不要有任何引用。
	4. D级内容指“需要详细了解”的内容，摘要压缩比例在1/5到1/15，不要有任何引用。
	5. E级内容指“知识，了解即可”，摘要压缩比例在1/10到1/20，不要有任何引用。
2. 所有询问中用户未提及的内容都是默认的 D 级内容
wiki/Index.md 示例格式说明：
1. `wiki/index.md` 应按月份分组，月份标题格式为 `## yyyy-MM`。
2. 示例：
	- `- [202604-纳瓦尔宝典｜最终摘要](sources/202604-纳瓦尔宝典.md) ｜ [实践计划](practice/202604-纳瓦尔宝典-实践计划.md)`
	- Source Path: `coreWiki/raw/2026/纳瓦尔宝典.pdf`
	- 简短摘要：`记录纳瓦尔的人生哲学，尤其是关于财富积累、判断力的增强、如何主动获取幸福、哲学等。`
3. 示例说明：
	- 第一行是该 source 的链接行；至少应包含最终摘要链接，如果存在实践计划，也应一并列出。
	- 第二行是 `Source Path`，必须写原始文件路径，并使用反引号包裹。
	- 第三行是 `简短摘要`，必须概括该条目的主要内容，供后续链接检查使用。
4. `简短摘要` 的长度要求必须严格执行（2026-07-03 修订）：
	- 无论内容长短，`简短摘要` 一律控制在**100字以内**。index 的职责是路由（"这是什么材料、什么主题该来查它"），不是缓存内容。
	- 压缩方式、模块拆分、双向链接统计、Confidence 详情等元信息**不进 index**，统一写在对应 `wiki/sources/*.md` 文件头部（第一个 `## ` 之前）的元信息 bullet 区。
5. 除示例外的其他 `index.md` 条目，也统一参考本模块的格式书写。
6. `index.md` 顶部维护一节 `## 主题索引`（紧跟一级标题之后）：把全库 sources 按主题聚成若干行，每行格式 `- 主题名：[文件短名](sources/xxx.md)、…`，整节不超过30行。新入库文档必须归入合适的主题行（没有合适主题时可新增一行）。query/probe 等命令应先读主题索引定位，再跳转条目。
7. 待入库清单只存放于 `wiki/uningest.md`，不得写入 `index.md`（见 `check_uningest.md`）。

### 实践计划
1. 如果没有涉及“A级内容”，那么“实践计划”部分不需要关注。
2. 必须根据A级内容中的东西，将描述的内容拆分为可实践/可练习的步骤，并制定为期14天~3个月之间的实践计划。实践计划的具体编写要求需要根据内容来确定，这部分需要询问用户。可以参考“询问规则”部分
3. 实践计划中每一次实践都必须包含最小可验证点，即用于检查是否已经达到当前目标点的checkpoint
4. 实践计划必须包含打分系统，在实践了一段实践之后，必须用可量化的结果来评价实践的成果。需要详细描述打分的规则。
5. 以上生成的内容尽量以原始资料为准，收束条件如果不足，尽量用选择题的形式提问用户，问题总数以3-10个为宜，如果10个问题无法收束，可以增加到20个问题。20个问题为硬性约束上限，但通常以3-10个为最佳。

## 新入库后的双向链接检查规范
1. 当一份新的原始材料完成摘要压缩任务后，必须再执行一次新入库后的双向链接检查；这里的“摘要压缩任务”包括普通文档与长文的一次摘要压缩。
2. 链接检查必须以 `wiki/index.md` 为入口，逐条检查已有条目的以下内容是否与新入库文档有关：
	- 链接行
	- `简短摘要`
3. 如果 `wiki/index.md` 中某个旧条目的链接行或 `简短摘要` 明显涉及与新入库文档相同、相近或直接相关的话题，就必须将该旧条目标记为“候选关联条目”。
4. 对每个“候选关联条目”，必须详细阅读对应的旧文档内容：
	- 旧的 `wiki/sources/*.md`
5. 同时也必须详细阅读新入库文档内容：
	- 新的 `wiki/sources/*.md`
6. 只有在详细阅读新旧文档之后，才可以决定是否建立链接；不能只凭 `index.md` 中的一行标题或一句摘要直接加链接。
7. 链接判断必须以二级标题为单位执行：
	- 必须提取新文档中所有以 `## ` 开头的二级标题，并逐个判断每个二级标题模块是否与候选旧文档存在真实关联
	- 也必须在旧文档中找到最相关的二级标题模块，用同样的二级标题粒度补回链
	- 不允许只在文档末尾做整篇级别的统一“相关条目”汇总
8. 如果确认某个二级标题模块与旧文档确实相关，则必须建立双向链接：
	- 在新入库文档的对应二级标题模块内部加入指向旧文档的链接
	- 在旧文档的对应二级标题模块内部加入指向新入库文档的链接
9. 双向链接必须在双方的 `wiki/sources/*.md` 中完成。
10. 链接格式必须使用普通 md 列表项，直接写在对应二级标题模块内部，不要额外添加“相关条目”“参考链接”等说明性标题。格式示例：`- [202604-穷查理宝典](202604-穷查理宝典.md)：两者都强调长期主义、独立判断、阅读与复利思维；本书更偏财富、幸福和自我塑造，《穷查理宝典》则更集中在跨学科思维模型、投资判断与误判心理学。`
11. 添加链接时，必须配上简短说明，写清楚为什么相关；不能只孤立地放一个裸链接。
12. 如果检查后发现并无真实关联，则不得为了追求链接数量而强行加链接。

## 长文压缩计划
1. 长文是指大于5000中文字的内容，或者大于2500个英文单词的内容。
2. 一旦判断为长文，必须立即切换到长文压缩流程，但仍然只做一次摘要压缩，不能再执行两次压缩。
3. 先完整读取原始资料，列出模块清单，并为每个模块分配唯一代号，例如：`一.财富 二.幸福 三.责任感`。
4. 列出模块后，必须先检查每个模块对应的原始材料长度。任何模块如果超过5000中文字，或主要使用英文且超过2500个英文单词，都必须在压缩前继续拆分为更小的子模块，并重新编号；在完成拆分前，不得进入压缩步骤。
5. 按“询问规则”向用户确认每个模块或子模块的重点等级；如果用户未回复、拒绝回答或无法提供，则将所有未定模块默认视为 D 级。
6. 在开始压缩之前，必须先向用户复述一遍每个模块或子模块的重点等级，以及该模块在摘要压缩中对应的压缩比例。
7. 在模块等级确认完成之后，必须先写出“长文压缩计划”。该计划至少应包含：
	- 模块顺序
	- 每个模块的代号和名称
	- 每个模块对应的原始材料范围
	- 每个模块对应的摘要压缩比例
	- 每个模块准备交给哪个 subagent 处理
	- 最终汇总目标文件，即 `wiki/sources/*.md`
8. 写完“长文压缩计划”后，才可以开始压缩。压缩必须按模块拆开，逐个把模块交给 subagent 执行；每个 subagent 只能负责自己被分配到的模块，不能跨模块合并压缩。
9. 压缩时，每个模块都必须严格遵守该模块在“重点等级划分”中对应的摘要压缩比例；不允许为了统一篇幅而改用整篇平均压缩比例。
10. 每个 subagent 完成压缩后，主 AI 必须回收各模块结果，检查是否遗漏模块、是否越界补充信息、是否违反引用限制，并逐一复核该模块的实际压缩效果是否与开头设定的摘要压缩比例一致；只有复核通过后，才可以按模块顺序汇总到同一个 `wiki/sources/*.md` 文件中。
11. 压缩汇总完成后，必须再启动一个独立的 subagent，专门执行文档关联的链接行为；该 subagent 只能负责检查关联、阅读相关旧文档与新文档，并按“新入库后的双向链接检查规范”补充双向链接，不能与摘要压缩任务混在同一个 subagent 中执行。
12. 文档关联链接 subagent 完成后，主 AI 才可以更新 `wiki/index.md`；如果存在实践计划，也应追加实践计划链接。
13. 如果“必须使用 subagent”的要求与当前运行环境的上层约束冲突，必须先按“硬性约束”中的冲突处理规则，显式说明冲突点并询问用户如何处理；在冲突未澄清前，不得擅自改为单 agent 执行。


## 硬性约束

- **必须询问每个章节的重点等级。不能按照你的主观意愿来划分重点**
- 只允许修改 `[libraryName]/wiki/` 下的文件。
- 不确定时必须标注 `Confidence: Low`。
- 禁止虚构来源、引用或 raw 路径。
- 无论是否为长文，每个模块对应的原始材料范围都不得超过5000中文字；如果模块主要使用英文书写，则对应的原始材料范围不得超过2500个英文单词；若超过该上限，必须先拆分模块，再进入压缩。
- 对于长文，必须严格遵守“长文压缩计划”。
- 新入库文档在完成全部摘要压缩任务后，必须严格遵守“新入库后的双向链接检查规范”。
- `wiki/sources/*.md` 中所有级别标题的标题内容都不得出现空格字符。
- 如果本规范与当前运行环境的上层约束冲突，AI 必须显式说明冲突点，并询问用户如何处理；在冲突未澄清前，不得假定本规范可以覆盖上层约束。
- 如果内容中有代码，那么代码内容**不能进行任何压缩**，应当选择重要的代码内容入库
